# Mosaic: CSS Modules vs Pigment CSS vs Tailwind CSS v4 vs Current Emotion System

## Executive Summary

All three Mosaic approaches eliminate runtime CSS-in-JS in favor of static CSS output. They differ in authoring model, build complexity, distribution model, and customization depth. This document compares all four systems across the dimensions that matter most for Clerk's UI library.

---

## Side-by-Side Comparison

| Dimension                    | Current (Emotion)                                                 | CSS Modules                                            | Pigment CSS                                                                   | Tailwind CSS v4                                                                                                      |
| ---------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------ | ----------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| **Style authoring**          | JS objects in component files via `css`/`sx` props                | Separate `.module.css` files                           | JS objects in component files via `styled()`                                  | Utility classes in JSX or `@apply` in CSS files                                                                      |
| **Style output**             | Runtime `<style>` injection                                       | Static CSS file                                        | Static CSS file                                                               | Static CSS file                                                                                                      |
| **Runtime cost**             | High — style computation + injection on every render              | Zero — class names only                                | Zero — class names only                                                       | Zero — class names only                                                                                              |
| **Bundle shipped to client** | Theme object + Emotion runtime (~12KB) + style computation code   | CSS file + class name strings                          | CSS file + class name strings                                                 | CSS file + class name strings                                                                                        |
| **Dark mode**                | JS theme swap (`dark` theme object)                               | `light-dark()` CSS function                            | `colorSchemes` + `getSelector` (class toggle)                                 | Class-based (`.dark`) or `prefers-color-scheme` media query — no `light-dark()` support                              |
| **Variant system**           | `createVariants` (custom, theme-aware)                            | `data-*` attributes + CSS selectors                    | Built-in `variants` array (prop-based, extracted)                             | Variant class maps in JS or `data-*` + `@apply` in CSS                                                               |
| **Design tokens**            | `InternalTheme` JS object ($-prefixed, runtime)                   | `--clerk-*` CSS custom properties                      | `--clerk-*` CSS custom properties (auto-generated by `extendTheme`)           | `--color-clerk-*` CSS custom properties (auto-generated by `@theme`) + utility classes                               |
| **Token definition**         | JS (`foundations/` + `parseVariables`)                            | Hand-authored `tokens.css`                             | JS theme object → auto-extracted to CSS vars                                  | `@theme` CSS directive → CSS vars + utility classes                                                                  |
| **User customization**       | `appearance` prop (variables, elements, themes)                   | CSS variable overrides + `[data-cl-slot]` targeting    | CSS variable overrides + optional `styleOverrides` via plugin                 | CSS variable overrides + `[data-cl-slot]` targeting; Tailwind consumers can also use `@source` for theme unification |
| **Customization depth**      | Deepest — 200+ element keys, state/id suffixes, CSS objects in JS | Moderate — CSS vars + external CSS targeting           | Moderate by default, deep with plugin (slot-level overrides)                  | Moderate — CSS vars + external CSS targeting (same as CSS Modules)                                                   |
| **Consumer setup**           | Zero (styles auto-inject)                                         | `import '@clerk/ui/mosaic.css'`                        | `import '@clerk/ui/mosaic.css'` (default) or Pigment plugin (advanced)        | `import '@clerk/ui/mosaic.css'` (default) or `@source` + `@import` (Tailwind consumers)                              |
| **Build tooling**            | None (runtime)                                                    | CSS Modules plugin in tsdown + `generateScopedName`    | Separate Vite build with Pigment plugin                                       | Separate Vite build with `@tailwindcss/vite` plugin                                                                  |
| **Colocation**               | Styles in same `.tsx` file                                        | Styles in separate `.module.css` file                  | Styles in same `.tsx` file                                                    | Styles in `.tsx` (utility classes) or separate `.module.css` (with `@apply`)                                         |
| **TypeScript integration**   | Full — theme types in `sx` callbacks                              | Partial — `styles` object typed via declaration file   | Full — `theme.vars.*` typed in style callbacks                                | None — utility class strings are untyped                                                                             |
| **Emotion coexistence**      | N/A (is the current system)                                       | Per-file `@jsxImportSource` pragma + separate tsconfig | Per-file `@jsxImportSource` pragma + separate tsconfig                        | Per-file `@jsxImportSource` pragma + separate tsconfig                                                               |
| **SSR / RSC compatibility**  | Requires Emotion SSR setup                                        | Works natively                                         | Works natively                                                                | Works natively                                                                                                       |
| **Library maturity**         | Emotion is stable, widely used                                    | CSS Modules is a web standard                          | Pigment CSS is v0.0.31 (alpha)                                                | Tailwind v4 released Jan 2025; `@theme`/`@reference` are new                                                         |
| **External dependency**      | `@emotion/react`, `@emotion/cache`                                | None (build tooling only)                              | `@pigment-css/react` (runtime dep) + `@pigment-css/vite-plugin` (build dep)   | `tailwindcss` + `@tailwindcss/vite` (build deps only)                                                                |
| **Modern CSS features**      | Requires library support for each feature                         | Full, unmediated access — it's real CSS                | Partial — JS object syntax can't express `@property`, `@layer`, `@scope` etc. | Partial — `@property`, `@scope`, `@starting-style` work in CSS files but not via utility classes                     |
| **Distribution risk**        | N/A                                                               | None — static CSS works everywhere                     | Medium — alpha dependency, plugin fragility                                   | High — `@layer` conflicts with consumer Tailwind (GitHub #17954, #18758)                                             |

## Authoring Experience

### Current (Emotion)

```tsx
// Styles colocated, theme-aware, but computed at runtime
const Button = styled('button')(({ theme }) => ({
  background: theme.colors.$primaryColor500,
  '&:hover': { background: theme.colors.$primaryColor600 },
}));
```

### CSS Modules

```css
/* Button.module.css — separate file */
.button[data-variant='solid'] {
  background: var(--clerk-color-primary);
}
.button[data-variant='solid']:hover {
  background: var(--clerk-color-primary-hover);
}
```

```tsx
// Button.tsx — clean, minimal
import styles from './Button.module.css';
<button
  className={clsx(styles.button, className)}
  data-variant={variant}
/>;
```

### Pigment CSS

```tsx
// Styles colocated, theme-aware, extracted at build time
export const Button = styled('button', {
  name: 'ClerkButton',
  slot: 'root',
})<ButtonProps>(({ theme }) => ({
  variants: [
    {
      props: { variant: 'solid' },
      style: {
        background: theme.vars.colors.primary,
        '&:hover': { background: theme.vars.colors.primaryHover },
      },
    },
  ],
}));
```

### Tailwind CSS v4

```tsx
// Utility classes inline in JSX
<button
  className={clsx(
    'gap-clerk-2 rounded-clerk-md font-clerk-sans inline-flex items-center justify-center font-medium',
    'bg-clerk-primary text-clerk-primary-contrast hover:bg-clerk-primary-hover',
    className,
  )}
/>
```

```css
/* Or: @apply in a CSS Module file via @reference */
@reference "tailwindcss";
.root[data-variant='solid'] {
  @apply bg-clerk-primary text-clerk-primary-contrast;
  &:hover {
    @apply bg-clerk-primary-hover;
  }
}
```

**Verdict**: Pigment CSS is closest to the current Emotion DX (colocated, JS objects, theme callbacks). Tailwind (utility classes in JSX) is the most familiar to developers who already use Tailwind. CSS Modules is a departure — styles live in separate files and use CSS syntax. Tailwind with `@apply` in CSS files is functionally a CSS Modules approach with an extra layer.

## Build Complexity

### Current (Emotion)

- No build step for styles — everything is runtime
- Existing tsdown + rspack pipeline handles JS only

### CSS Modules

- Add CSS Modules plugin to tsdown config
- Configure `generateScopedName` to auto-prepend `cl-`
- Concatenate `tokens.css` + compiled modules into `mosaic.css`
- Copy to dist and expose via package exports
- **Complexity: Low** — CSS Modules are a well-understood, stable standard with broad tooling support

### Pigment CSS

- New Vite config (`vite.mosaic.config.ts`) separate from existing tsdown pipeline
- Pigment plugin processes `styled()` calls at build time via WyW-in-JS evaluation
- Theme object must be importable and evaluable in Node context
- Dual output: pre-extracted default + raw source for plugin users
- `extendTheme` generates CSS variable declarations automatically
- **Complexity: Medium** — adds a second build tool (Vite alongside tsdown), depends on alpha-stage library internals

### Tailwind CSS v4

- New Vite config with `@tailwindcss/vite` plugin
- `@theme` directive defines tokens (CSS vars + utility classes in one step)
- Tailwind compiles utilities at build time, outputs static CSS
- Must handle `@layer` conflict if consumer also uses Tailwind — no clean solution exists
- Two distribution paths (pre-compiled CSS vs source) require separate testing
- **Complexity: Medium** — Tailwind itself is mature, but the library distribution model is unresolved

**Verdict**: CSS Modules wins on build simplicity. Tailwind and Pigment CSS both require a separate Vite build. Tailwind adds the unique complication of `@layer` conflicts with consumer Tailwind installations.

## Customization Power

### Current (Emotion)

- `variables`: 15-shade color scales auto-generated from a single input
- `elements`: 200+ targetable keys with state/id suffixes, CSS objects or class strings
- `theme`: Composable prebuilt themes (dark, shadcn, neobrutalism)
- Cascade: global → component-specific → element-level, with recursive theme expansion
- **Power: Maximum** — but at the cost of significant complexity and runtime overhead

### CSS Modules

- Override `--clerk-*` CSS custom properties on `:root`
- Target `cl-*` class names + `data-*` attributes in user CSS
- No JS API for customization
- **Power: Moderate** — covers 90% of use cases (colors, spacing, fonts, individual component overrides) but can't match the granularity of 200+ element keys with state variants

### Pigment CSS

- Default path: same as CSS Modules (CSS variable overrides)
- Advanced path: `styleOverrides` per component slot via Pigment plugin — closest to Emotion's `elements` API but requires consumer-side build setup
- **Power: Moderate-to-High** — default path is simple, advanced path recovers much of Emotion's depth for users willing to add the plugin

### Tailwind CSS v4

- Default path: same as CSS Modules (CSS variable overrides + `[data-cl-slot]` targeting)
- Tailwind consumers can import Clerk's `@theme` tokens and use them as utility classes in their own components
- No equivalent to Pigment's `styleOverrides` — customization depth is the same as CSS Modules
- **Power: Moderate** — same as CSS Modules for external customization, plus utility class reuse for Tailwind consumers

**Verdict**: If deep customization for every element/state combination is essential, Emotion is still unmatched. Pigment CSS's advanced path recovers some of this. CSS Modules and Tailwind v4 are equivalent for consumer customization — both offer CSS variables + selector targeting. Tailwind adds utility class reuse for consumers who already use Tailwind, but this doesn't increase customization depth. The key question is whether the 200+ element keys are actually used by customers — if most users only override colors/fonts/spacing, CSS variables are sufficient.

## Risk Profile

### CSS Modules

| Factor               | Risk Level                                                                 |
| -------------------- | -------------------------------------------------------------------------- |
| Technology maturity  | **Very Low** — CSS Modules is a stable web standard, 10+ years old         |
| Dependency risk      | **None** — no new runtime dependencies                                     |
| Build fragility      | **Low** — well-understood tooling, simple config                           |
| Breaking change risk | **Very Low** — CSS is stable, class names are deterministic                |
| Team adoption        | **Medium** — requires shift from colocated JS styles to separate CSS files |

### Pigment CSS

| Factor               | Risk Level                                                                                                    |
| -------------------- | ------------------------------------------------------------------------------------------------------------- |
| Technology maturity  | **High** — v0.0.31 alpha, API subject to breaking changes                                                     |
| Dependency risk      | **Medium** — `@pigment-css/react` becomes a runtime dependency                                                |
| Build fragility      | **Medium** — WyW-in-JS evaluation can fail on non-trivial expressions, closures in variants are not supported |
| Breaking change risk | **High** — 0.0.x versioning, no stability guarantees                                                          |
| Team adoption        | **Low** — similar DX to Emotion, familiar patterns                                                            |

### Tailwind CSS v4

| Factor               | Risk Level                                                                                    |
| -------------------- | --------------------------------------------------------------------------------------------- |
| Technology maturity  | **Low** — Tailwind is mature and widely used; v4's `@theme`/`@reference` are newer (Jan 2025) |
| Dependency risk      | **Low** — build dependency only, not shipped to consumers                                     |
| Build fragility      | **Medium** — `@layer` conflicts with consumer Tailwind are a documented, unsolved issue       |
| Breaking change risk | **Low** — Tailwind follows semver, v4 is stable                                               |
| Distribution risk    | **High** — no clean path that works for both Tailwind and non-Tailwind consumers              |
| Team adoption        | **Low** — Tailwind is widely known, minimal learning curve                                    |

**Verdict**: CSS Modules is the safest — zero dependency risk, zero distribution risk. Tailwind v4 is stable as a tool but has an unsolved distribution problem for component libraries. Pigment CSS carries the highest risk as an alpha dependency.

## Performance

All three Mosaic approaches are equivalent at runtime — zero JS style computation, just static CSS + class name strings. The differences are:

| Metric         | Current (Emotion)                                     | CSS Modules                    | Pigment CSS                    | Tailwind CSS v4                |
| -------------- | ----------------------------------------------------- | ------------------------------ | ------------------------------ | ------------------------------ |
| First paint    | Slower (style computation blocks render)              | Faster (CSS loaded upfront)    | Faster (CSS loaded upfront)    | Faster (CSS loaded upfront)    |
| Re-render cost | Style recomputation on theme/prop changes             | Zero (class toggle only)       | Zero (class toggle only)       | Zero (class toggle only)       |
| JS bundle size | Larger (Emotion runtime + theme object)               | Smaller (no style runtime)     | Smaller (no style runtime)     | Smaller (no style runtime)     |
| CSS payload    | Inline `<style>` tags (can't be cached independently) | Static `.css` file (cacheable) | Static `.css` file (cacheable) | Static `.css` file (cacheable) |
| SSR            | Requires Emotion SSR extraction                       | Works natively                 | Works natively                 | Works natively                 |

**Verdict**: Tie — all three Mosaic approaches are equally fast at runtime. The win over Emotion is significant in all cases.

## Modern CSS Feature Access

CSS evolves independently of any JS library. The question is: how quickly can each approach adopt new CSS features?

| CSS Feature                                        | Current (Emotion)                             | CSS Modules | Pigment CSS                                                     |
| -------------------------------------------------- | --------------------------------------------- | ----------- | --------------------------------------------------------------- |
| `@property` (animatable tokens, type-checked vars) | Not supported in JS objects                   | **Native**  | Not supported in JS objects — requires `globalCss` escape hatch |
| `light-dark()`                                     | Not supported                                 | **Native**  | Not supported — uses `colorSchemes` instead                     |
| `@layer` (cascade ordering)                        | Partially supported via `cssLayerName` option | **Native**  | Not supported in JS objects                                     |
| `@scope` (style scoping)                           | Not supported                                 | **Native**  | Not supported in JS objects                                     |
| `@container` (container queries)                   | Supported via string keys                     | **Native**  | Supported via string keys                                       |
| `@starting-style` (entry animations)               | Not supported                                 | **Native**  | Not supported in JS objects                                     |
| Native CSS nesting                                 | Not applicable (uses JS nesting)              | **Native**  | Partially supported via `&` in style objects                    |
| `:has()` selector                                  | Supported via string keys                     | **Native**  | Supported via string keys                                       |
| `color-mix()`                                      | Supported in string values                    | **Native**  | Supported in string values                                      |

### Why this matters for a design system

**`@property` is the standout.** Registering color tokens with `@property` enables smooth animated transitions between themes — colors interpolate instead of snapping. This is impossible with unregistered CSS custom properties.

```css
/* Register the token */
@property --clerk-color-primary {
  syntax: '<color>';
  inherits: true;
  initial-value: #6c47ff;
}

/* Now this transition actually works */
.cl-button {
  background: var(--clerk-color-primary);
  transition: background 0.3s ease;
}
```

Without `@property`, changing `--clerk-color-primary` causes an instant snap. With it, the browser interpolates between the old and new color values. This is a premium design system feature that's only available when writing real CSS.

**Future-proofing:** When new CSS features ship in browsers, CSS Modules can use them immediately. Emotion and Pigment CSS must add explicit support for each feature — their JS object syntax is a fixed vocabulary that doesn't automatically grow with CSS. As CSS gains more @-rules and functions, the gap widens.

**Verdict**: CSS Modules has a structural advantage — it IS CSS, so it gets every CSS feature for free. Pigment CSS and Emotion are JS abstractions over CSS, and every new CSS feature requires library work. For a design system that will evolve over years, this is a significant long-term advantage.

## Migration Path from Current System

### CSS Modules

- **Clean break**: Different authoring model (CSS files vs JS objects), different variant system (`data-*` vs `createVariants`), no theme callbacks
- Existing component styles cannot be mechanically converted — they must be manually rewritten in CSS
- No shared infrastructure between old and new systems
- The gap between authoring models means the team maintains two distinct mental models during coexistence

### Pigment CSS

- **Gradual migration possible**: Same authoring model (JS style objects), similar variant concept, theme callbacks still work
- Existing Emotion `styled()` calls could be incrementally migrated to Pigment's `styled()` with moderate effort
- Theme structure can be shared (Pigment's `extendTheme` accepts a similar shape)
- The authoring similarity means less context-switching during the coexistence period

**Verdict**: Pigment CSS offers a smoother migration path from the current Emotion system. CSS Modules is a bigger paradigm shift.

## Recommendation

### Choose CSS Modules if:

- You want the **lowest risk** and **simplest architecture**
- You're comfortable with the authoring shift to separate CSS files
- Deep customization (200+ element keys) is not needed — CSS variables cover your use cases
- You want **zero new runtime dependencies**
- You value long-term stability over migration convenience
- You want **unmediated access to modern CSS** — `@property` for animatable tokens, `@layer`, `@scope`, `@starting-style`, and every future CSS feature without waiting for library support

### Choose Pigment CSS if:

- You want to **preserve the colocated JS authoring DX** the team already knows
- You need the **advanced customization path** (component slot overrides via plugin) for power users
- You're comfortable with an **alpha dependency** for an exploration
- Gradual migration from Emotion matters — you want shared patterns between old and new
- You believe MUI's backing gives Pigment CSS sufficient longevity

### The trade-off in one sentence

**CSS Modules trades authoring familiarity for stability, future-proofing, and full CSS language access; Pigment CSS trades all three for authoring familiarity.**

## What Both Approaches Share (vs Current System)

Regardless of which Mosaic approach is chosen, the following improvements apply equally:

- **No runtime CSS computation** — styles are static, improving paint performance and eliminating re-render style costs
- **Cacheable CSS** — a static `.css` file that browsers can cache independently, vs inline `<style>` tags
- **Smaller JS bundles** — no Emotion runtime, no theme object shipped to the client
- **Simpler mental model** — no appearance cascade, no `parseAppearance` pipeline, no `makeCustomizable` HOC
- **Native SSR/RSC support** — no special extraction needed for server rendering
- **CSS variable customization** — users override `--clerk-*` custom properties in plain CSS, no JS API needed
- **Semantic tokens** — purpose-driven token names (`primary-hover`, `fg-muted`) instead of numbered scales (`$primaryColor600`)
- **Single CSS import** — `import '@clerk/ui/mosaic.css'` replaces the current zero-config-but-runtime approach
